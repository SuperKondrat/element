## 1. Что за задача

Junior Fullstack тестовое для девелоперской компании. Нужен сервис над каталогом лотов (квартир):

- **Публичная витрина** — список лотов с фильтрами (ЖК, комнатность, диапазон цены за м², статус), сортировкой, пагинацией; бронь лота и заявка через форму (контакт + комментарий).
- **Админка** — вход (superuser из конфига, без регистрации), загрузка XML-фида (создаёт набор лотов), выбор активного набора, просмотр броней и заявок.

Оценивается инженерное суждение (выбор подходов, обоснование, чистые границы слоёв), а не объём кода — это НЕ олимпиадная задача, и предполагается решение с ИИ.

## 2. Модель данных

- **Лот**: `id` (PK), `external_id`, `set_id` (FK → набор), проект/ЖК, адрес, комнатность, площадь (м²), этаж, цена, цена базовая, статус (`в продаже`/`забронирован`/`продано`), `created_at`, `updated_at`. Цена за м² — вычисляемое поле, не хранится.
- **Набор лотов**: `id` (PK), имя/исходный файл, `uploaded_at`, количество лотов, признак активного (только один набор активен, показывается в публичной витрине).
- **Заявка**: `id` (PK), `lot_id` (FK, опционально), контакт (имя, телефон/почта), свободный комментарий, статус (`новая`/`в работе`/`закрыта`), `created_at`.
- **Бронь**: `id` (PK), `lot_id` (FK, обязательно), контакт, статус (`активна`/`отменена`), `created_at`. Бизнес-правило: бронировать можно только лот в статусе «в продаже»; после брони лот → «забронирован»; повторно забронировать нельзя.

Модель своя, парсится из фида — повторять структуру `element.xml` не обязательно.

## 6. Модель конфигурации (согласовано)

Требование ТЗ (п.4.3): секреты отделены от техпараметров; изменение одного значения (например, порта БД) не требует правок в нескольких местах; плоский `.env` со всем вперемешку и задвоенными значениями — не годится.

Решение — два файла:

| Файл | Git | Содержит | Примеры переменных |
|---|---|---|---|
| `.env` (из `.env.example`) | игнорируется | секреты, которые задаёт пользователь | `POSTGRES_PASSWORD`, `ADMIN_USERNAME`, `ADMIN_PASSWORD`, `JWT_SECRET_KEY` |
| `config/app.env` | коммитится | технические параметры с осмысленными дефолтами | `POSTGRES_HOST=db`, `POSTGRES_PORT=5432`, `POSTGRES_DB=element`, `POSTGRES_USER=element`, `LOG_LEVEL=info`, `JWT_EXPIRE_MINUTES=60`, `CORS_ORIGINS=...` |

Механика:
- Оба сервиса в `docker-compose.yml` подключают оба файла: `env_file: [config/app.env, .env]`.
- Каждый параметр существует **ровно в одном месте**. Порт БД — переменная `POSTGRES_PORT` в `config/app.env`, используется и в маппинге `ports:` сервиса `db`, и передаётся в `backend` — меняешь один раз, подхватывается везде по имени переменной.
- `DATABASE_URL` нигде не хранится готовой строкой — `Settings` (pydantic-settings) собирает её на лету из частей (`HOST/PORT/USER/PASSWORD/DB`). Хранение готовой строки отдельно = порт задваивается — ровно та ловушка, о которой предупреждает ТЗ.

**Нюанс для отчёта:** Docker Compose делает `${VAR}`-подстановку в самом YAML (например, `ports: - "${POSTGRES_PORT}:5432"`) только из корневого `.env`, а не из произвольных `env_file:`. Поэтому `POSTGRES_PORT` физически лежит в `.env`, хотя не является секретом — это осознанный, проговорённый в отчёте компромисс, а не недосмотр.

## 8. Решённые вопросы дизайна (обоснование — пригодится на защите)

### 8.1 Форма JSON-RPC dispatcher

**Решение:** один эндпоинт `POST /rpc`, роутинг по полю `method` (`lots.list`, `bookings.create`, ...), именованные `params` (объект, не массив), доменные ошибки — в зарезервированном диапазоне JSON-RPC `-32000…-32099`.

**Почему:**
- Несколько эндпоинтов под RPC (`/rpc/lots`, `/rpc/bookings`) нарушили бы спецификацию — JSON-RPC 2.0 допускает batch-запрос (массив вызовов в одном теле), а батч нельзя раскидать по разным URL. Один эндпоинт — не стиль, а требование протокола.
- Точечная нотация в имени метода (`lots.list`) читается как неймспейс ресурса и один-в-один мапится на REST-роут — удобно для сравнения протоколов в отчёте.
- Именованные `params` валидируются той же Pydantic-моделью, что и REST body — не нужно поддерживать два разных способа валидации одних и тех же данных.
- И REST, и RPC — тонкие адаптеры над одним `services/`: доменная ошибка (например, `LotAlreadyBookedError`) бросается один раз, каждый протокол сам решает, во что её завернуть (HTTP 409 vs RPC `-32001`). Это и есть чистая граница между слоями, которую просят показать в критериях.

### 8.2 Переключатель транспорта на фронте

**Решение:** глобальный тумблер в шапке UI, реализован как React Context + localStorage (хук `useTransport()`). `RestApiClient` и `RpcApiClient` реализуют один и тот же TS-интерфейс, `pages/`/`components/` дергают только `useApi()`, не зная, какой транспорт активен.

**Почему:**
- Локальный тумблер на каждой странице/форме создаёт UI-шум и на защите пришлось бы переключать транспорт в нескольких местах вместо одного — хуже демонстрируется, реальной пользы не даёт.
- Переключение на этапе сборки (`VITE_TRANSPORT` в `.env`, ребилд) не подходит формально: ТЗ требует именно «переключатель», то есть рантайм-переключение без пересборки.
- Только Context (без localStorage) сбрасывает выбор на `F5` — рискованно посреди защиты, если страница перезагрузится. localStorage просто переживает reload, Context остаётся реактивным источником истины внутри дерева компонентов.
- Общий TS-интерфейс для обоих клиентов — та же демонстрация границ слоёв, что и на бэкенде, только на фронте.

### 8.3 Маппинг полей фида → модель лота

**Решение:** строгий парсинг — берём из `element.xml` только то, что требует модель п.3.2 ТЗ, остальное отбрасываем, сырой XML/JSON не сохраняем.

Проверено по реальному `element.xml` (`grep` по всему файлу): у `<flat>` статус принимает ровно три значения — `FREE` / `RESERVED` / `SOLD`, что один-в-один ложится на `в продаже` / `забронирован` / `продано` без доп. случаев.

| Поле модели | Тег в фиде | Комментарий |
|---|---|---|
| `external_id` | `flat_id` | uuid-подобный, стабильнее чем `id_old` между перезаливками фида |
| комнатность | `room` (+ `size.name` как человекочитаемый дубль) | `room=0` похоже на студию — свериться на паре примеров при реализации парсера |
| площадь | `area` | `living_area`/`kitchen_area` есть в фиде, но моделью не требуются |
| этаж | `floor` | прямое соответствие |
| цена / цена базовая | `price` / `price_base` | у части лотов `price=0` — обычный edge case для фильтра по цене за м² |
| статус | `status` | `FREE`/`RESERVED`/`SOLD` → `в продаже`/`забронирован`/`продано` |
| проект/ЖК, адрес | `object.name`, `object.address` | берём с объекта (не с корпуса — корпус дублирует адрес объекта) |

**Осознанно игнорируем:** `images`, `description_main`, `infrastructure`, `developer`, `profits_main`, `discounts`, `sales_info`, `videos` на уровне объекта; `axis`, `porch`, `column*`, `azimuth`, `height`, `tech_number`, `balconies`, `plans`, `window_view`, `bathroom`, `renovation`, `decoration`, `description` на уровне квартиры.

**Почему:**
- Модель из п.3.2 ТЗ их не требует — тащить «про запас» это расширение фич без запроса, а критерии прямо штрафуют «объём фич в ущерб качеству одного среза».
- Альтернатива — хранить весь `<flat>` в jsonb-колонке про запас — добавляет неиспользуемую сложность без выигрыша.
- Если на защите попросят живой пример «добавить новую операцию / расширить модель» (п.10 ТЗ) — можно на месте показать добавление, например, поля `renovation` из уже игнорируемого набора: готовый и понятный пример расширяемости без домашней заготовки.

### 8.4 Auth-транспорт админки

**Решение:** Bearer JWT в заголовке `Authorization`, хранится на фронте в localStorage. Один и тот же токен работает и для REST, и для JSON-RPC вызовов.

**Почему:**
- httpOnly cookie требовала бы отдельной настройки CORS/SameSite и CSRF-защиты сразу для двух протоколов (REST и JSON-RPC) — лишняя сложность не по делу задачи.
- Bearer-токен — единственный механизм авторизации, не завязанный на транспорт: просто заголовок, одинаково добавляется в оба клиента (`RestApiClient`/`RpcApiClient`).
- XSS-риск localStorage для этой задачи не критичен: регистрации и ролевой модели нет (п.7 ТЗ прямо исключает их из оценки), в системе один superuser из конфига — не пользовательские данные под угрозой.
